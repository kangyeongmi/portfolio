# UX Writing Guidelines

## 1. 목적
이 문서는 티맵모빌리티 지원용 포트폴리오 웹페이지의 문구 기준을 정리한다.  
목표는 채용자가 적은 노력으로 핵심 역량과 프로젝트 맥락을 이해하게 만드는 것이다.

## 2. 기본 원칙
- 짧게 쓴다.
- 한 문장에는 하나의 메시지만 담는다.
- 평가표 같은 표현을 줄인다.
- 사용자가 한 번 더 생각해야 하는 말을 피한다.
- 확인되지 않은 성과는 쓰지 않는다.
- 내가 하지 않은 역할을 한 것처럼 쓰지 않는다.
- "모빌리티 경험"이 아니라 "모빌리티 UX로 확장 가능한 경험"이라고 쓴다.

## 3. 말투
- 기본은 담백한 해요체 또는 간결한 서술형을 사용한다.
- 섹션 제목은 명사형으로 짧게 쓴다.
- 본문은 너무 격식 있게 쓰지 않는다.
- 자기소개 문장은 과장하지 않는다.

## 4. 피해야 할 표현
| 피할 표현 | 이유 | 대체 표현 |
|---|---|---|
| 티맵에서 볼 역량 | 평가표처럼 보임 | 할 수 있는 일 |
| 내 경험의 증거 | 딱딱함 | 해본 일 |
| 연결 프로젝트 | 문서 느낌이 강함 | 프로젝트 |
| 모빌리티 도메인 경험 보유 | 과장 위험 | 모빌리티 UX로 확장 가능한 경험 |
| 사용성 향상 | 근거 부족 | 어떤 행동이 쉬워졌는지 쓰기 |
| UI 개선 | 범위가 모호함 | 정보 구조를 바꿨다 / 흐름을 다시 정리했다 |
| 데이터 기반 판단 UX | 추상적 | 데이터를 보고 바로 판단하게 만들었다 |

## 5. 권장 표현
| 상황 | 권장 문장 |
|---|---|
| 현장 UX | 현장에서 바로 판단하게 만들었어요 |
| Task Flow | 복잡한 흐름을 행동 단위로 다시 정리했어요 |
| AI UX | 필요한 정보를 빠르게 찾게 만들었어요 |
| Data UX | 데이터를 보고 다음 행동을 판단하게 만들었어요 |
| 도메인 전환 설명 | 내비게이션을 직접 설계한 경험은 아직 없습니다. 대신 현장 사용성, 복잡한 흐름, AI 정보 탐색, 데이터 화면을 설계해왔습니다. |

## 6. 섹션별 문구 규칙
### Hero
- 한 문장으로 포지셔닝을 말한다.
- "8년차", "복잡한 흐름", "빠른 판단"을 중심으로 쓴다.

### Recruiter Summary
- 평가표처럼 쓰지 않는다.
- "역량/증거/연결"보다 "할 수 있는 일/해본 일/프로젝트"로 쓴다.

### Project Card
- 프로젝트명, 한 줄 요약, 태그만 남긴다.
- 버튼 설명은 없앤다.
- 카드 자체가 클릭 가능해야 한다.

### Project Detail
- Problem, Strategy, Ownership, Evidence 순서로 쓴다.
- 문장마다 내가 한 일과 근거가 보여야 한다.

## 7. 최종 체크리스트
- 문장이 2줄 이상 길어지지 않는가?
- "사용성 향상"처럼 모호한 말이 없는가?
- 확인되지 않은 수치가 없는가?
- 채용자가 한 번에 이해할 수 있는가?
- 기획서가 아니라 포트폴리오처럼 읽히는가?
